## Plan: Members Directory UX V2

### Vision

Replace the current card-list + inline-form layout with three purpose-built views:
1. **Members Directory** — dense table, KPI strip, row-level actions.
2. **Registration Panel** — 4-step guided slide-over (administrative checklist → fare selection → accounting preview → committee assignment), triggered from a directory row action.
3. **Committees Management** — dedicated route (`/commissions`) with a committee card grid and an administrative overview table; first-class nav item, not a tab.

The design follows the v2 wireframe (Processus d'Adhésion v2 + Annuaire des Membres v2 + Gestion des Commissions).

---

### Layout Overview (from wireframe)

**Left panel – Members Directory**
- Page header: "Members Directory" title + subtitle, `Export CSV` + `Add New Member` buttons (top-right).
- KPI strip (4 tiles): Total Members | Pending Renewals (due by EOFY, highlighted in orange) | Active Instructors | Guest/Temp Passes.
- Filter bar: full-width text search (by name, ID, email) + quick filters (Category, Role, Can Fly) + advanced filter toggle icon.
- Density table with columns: NAME & ID | CATEGORY | ROLE FLAGS | OPERATIONAL STATUS | REGISTRATION | COMMISSION | ACTIONS.
  - NAME & ID: avatar/initials circle, full name, account_id in monospace sub-label.
  - CATEGORY: plain text (Full Member, Temp Member, Student…).
  - ROLE FLAGS: colored inline tags (INSTRUCTOR, EMPLOYEE, TOW PILOT, WINCH DRIVER) or "NO ROLES" muted text.
  - OPERATIONAL STATUS: colored badge (Active green, Suspended red, …).
  - REGISTRATION: icon + text badge (✅ COMPLETED, ⏳ DRAFT) year-aware.
  - COMMISSION: compact badge(s) showing assigned committee code(s) for the current year (e.g. SAFETY, SOCIAL). Empty if unassigned.
  - ACTIONS: pencil icon (edit form drawer) + kebab menu (Finalize Registration, View Sheet, …).
  - Warning indicator: ⚠ triangle beside name for members with renewal issues; optional left-border red/orange stripe on row.
- Pagination footer: "Showing X to Y of N members" + page buttons.
- Directory legend strip: color meanings for status and registration badges.

**Registration panel (slide-over)**
- Opens when "Finalize Registration" is triggered from a row action.
- Member header: avatar, full name, badge (e.g. RENEWAL REQUIRED), "Member since [date]", "Last Activity" info.
- Step 1 – Administrative Checklist: scrollable checklist items each with status chip (✅ VALID / ⚠ PENDING INPUT / ✅ VERIFIED). Items: Medical Certificate (Class 2), FFVP License Number, Identity Document. Informational only in V2 (not blocking backend).
- Step 2 – Fare Selection: table with columns Description | Category | Amount | Select (checkbox). Rows come from the pricing module. Shows TOTAL AMOUNT DUE in blue.
- Step 3 – Accounting Preview: read-only table (ACCOUNT | DEBIT | CREDIT) with the draft GL entry auto-generated by fare selection. Shows Invoice Reference.
- Step 4 – **Choix de Commission** (MANDATORY): "Members are required to participate in at least one working committee to support club operations." Renders active committee cards (icon + name, e.g. Maintenance, Events, Safety) as a selectable grid. At least one must be selected before VALIDATE is enabled.
- Footer: CANCEL | Effective Date input (defaults to today) | **VALIDATE & ACTIVATE REGISTRATION** (disabled until Steps 2 and 4 are satisfied).

**Committees Management page (`/commissions`)**
- Standalone routed page, separate nav item ("COMMISSIONS") in the shell sidebar.
- Page header: "Committees Management" + subtitle + `Add Committee` button (top-right).
- **Card grid** (top section): one card per active committee showing name, description, active member count, member avatars (first 3–4), "Manage Roster" / "View All" action, status badge, and any pending alert badge (e.g. "5 Pending Reports").
- **Administrative Overview table** (bottom section): columns Code | Committee Name | Manager | Last Meeting | Budget Status | Actions. Export CSV + Print Summary buttons above the table.
- `+ New Entry` / `Add Committee` action creates a new committee via existing `useCreateCommitteeMutation`.

### Implementation Phases

#### Phase 1 – Spec alignment and scope lock ✅
- Updated `docs/SPEC_MEMBERS.md` with the "Members UX V2 – Directory First" section covering:
  - Dense table layout with all columns listed above.
  - Mandatory accounting template / fare selection in registration flow.
  - Read-only `is_active` field governance.
  - Acceptance criteria for row actions, KPI strip, filter/query matrix, year-aware registration badges.
  - Committees Management page spec at `/commissions`.
  - Backend changes required table (committee_codes, atomic committee assignment, last_meeting_date).
- `MembersListPage` confirmed as the canonical `/members` component; `MembersPage` is migration source only.

#### Phase 2 – KPI strip + table skeleton (depends on Phase 1) ✅
- Replace the card-button list in `MembersListPage.tsx` with a proper `<table>` / shadcn `DataTable`-style layout.
- Add KPI strip computed from list query data:
  - **Total Members**: `members.length` (or total from pagination metadata if available).
  - **Pending Renewals**: members where `last_registration_year < selectedYear && is_active`.
  - **Active Instructors**: members where `is_instructor && is_active`.
  - **Guest/Temp Passes**: members where `member_category === 2` (Temporary).
- Render NAME & ID with initials avatar (derived from first/last name, no photo upload needed in V2).
- Render ROLE FLAGS as compact colored tags using existing `is_instructor`, `is_employee`, `is_executive`, `is_board_member` fields.
- Render OPERATIONAL STATUS badge mapped from `status` field.
- Render REGISTRATION badge mapped from `has_member_sheet_for_year` + `registration_status`, year-aware.
- Add ⚠ warning indicator for members with `last_registration_year < selectedYear && is_active`.
- Keep desktop-first density; collapse role flags and registration column on mobile.

#### Phase 3 – Registration slide-over panel (depends on Phase 2)
- Create `RegistrationPanel.tsx` component (slide-over drawer pattern using shadcn Sheet).
- Panel triggered by "Finalize Registration" in row kebab menu.
- **Step 1 – Administrative Checklist**: render checklist items with status derived from member data (ffvp_id presence, etc.). Informational only in V2.
- **Step 2 – Fare Selection**: fetch pricing items via existing pricing query (reuse `usePricingVersionsQuery` or equivalent); render as selectable table with checkboxes. Compute TOTAL AMOUNT DUE client-side using `decimal.js`. At least one item must be selected.
- **Step 3 – Accounting Preview**: derive preview GL lines from selected fare items + member account mapping. Display read-only. Generate an Invoice Reference preview string.
- **Step 4 – Committee Selection (MANDATORY)**: fetch active committees via `useCommitteesQuery(activeOnly: true)`; render as a selectable card grid (icon, name, short description). At least one must be selected. On submit, call `useReplaceCommitteeMembersMutation` for each selected committee after registration completes.
- VALIDATE button disabled until Step 2 (fare selected) AND Step 4 (committee selected) are both satisfied.
- Footer: Effective Date (date input, defaults to today), CANCEL (close panel), VALIDATE & ACTIVATE REGISTRATION (calls `useCompleteRegistrationMutation` with `{ year, template_uuid, selected_items[] }` then committee roster mutations sequentially).
- On success: close panel, invalidate member list + committees queries, update row REGISTRATION and COMMISSION badges without full reload.

#### Phase 4 – Filter expansion + advanced filter panel (parallel with Phase 3)
- Add advanced filter drawer triggered by the filter icon in the filter bar.
- Expose: `registration_status`, `is_active`, role flags (`is_instructor`, `is_employee`, `is_executive`, `is_board_member`), `committee_uuid`, `year`.
- Rule: `registration_state` filter requires `year` to be set; show inline guidance if year is missing.
- Keep quick filters (Category, Role, Can Fly) in the always-visible bar.
- Surface backend filter errors clearly in the filter bar.

#### Phase 5 – Form governance and lifecycle locking (depends on Phase 3)
- In `membersShared.tsx`: make `is_active` field render as a read-only display, not an editable checkbox.
- Member edit form (pencil icon → drawer/sheet): show `is_active` as a locked status badge.
- Ensure no UI path allows toggling `is_active` directly; lifecycle changes happen only via registration/business flows.

#### Phase 6 – Pagination (depends on Phase 2)
- Add pagination footer: "Showing X to Y of N members".
- If backend supports `limit`/`offset` or `page`/`size` query params, wire up page navigation.
- If backend returns the full list, implement client-side pagination (page size 25).
- Check `MemberFilters` and backend route for pagination param support first.

#### Phase 7 – Committees Management page (can start after Phase 1)
- Create `CommitteesManagementPage.tsx` as a standalone routed component at `/commissions`.
- **Card grid section**: map over `useCommitteesQuery()` data; each card shows code badge, name, description, active member count, member avatar stack (first 3–4), status badge, alert badge if applicable.
  - "Manage Roster" button opens `CommitteeRosterDrawer.tsx` (member multi-select, saves via `useReplaceCommitteeMembersMutation` — migrated from `MembersPage`).
- **Administrative Overview table**: `<table>` with Code | Committee Name | Manager | Last Meeting | Budget Status | Actions.
  - Export CSV: generate client-side from query data.
  - Print Summary: `window.print()`.
- `Add Committee` button opens a create committee form drawer (migrate form logic from `MembersPage`).
- Add `COMMISSIONS` nav entry in the shell sidebar pointing to `/commissions`.
- Add route in `App.tsx`: `<Route path="/commissions" element={<CommitteesManagementPage />} />`.

#### Phase 8 – Consolidation and cleanup (depends on Phases 2–7)
- Migrate any remaining useful logic from `MembersPage.tsx` to `MembersListPage.tsx`, `CommitteesManagementPage.tsx`, or `membersShared.tsx`.
- Remove or stub out `MembersPage.tsx` once all three target pages cover its functionality.
- Single source of truth for member form primitives remains `membersShared.tsx`.
- Ensure `App.tsx` route for `/members` still points to `MembersListPage`.

### Decisions
- Table density: shadcn DataTable-style `<table>` replacing card-button list.
- KPI source: compute from loaded list data client-side (fast path); add backend aggregate endpoint later at scale.
- Row actions UX: pencil icon for edit + kebab menu for secondary actions (Finalize Registration, View Sheet).
- Registration panel: 4-step right slide-over (Sheet component), not a modal — allows reading the directory while the panel is open.
- Step 4 (Committee) is **MANDATORY** — VALIDATE button stays disabled until at least one committee card is selected AND at least one fare is selected.
- Committee assignment in Step 4 calls `useReplaceCommitteeMembersMutation` after `useCompleteRegistrationMutation` succeeds.
- COMMISSION column in directory: show committee code badges from `committee_codes` on `MemberSummary`. If backend doesn't expose `committee_codes` yet on the list endpoint, show `committee_count > 0` as a placeholder and track as a backend task.
- Committees Management at `/commissions` replaces the committees section that currently lives inside `MembersPage`; it is a standalone nav item, not a sub-section.
- Template / fare selection: mandatory in registration panel before VALIDATE is enabled.
- `is_active` governance: read-only display everywhere in V2.
- Legacy cleanup: remove `MembersPage` dead paths in Phase 8, not behind a feature flag.

### Open questions / backend tasks
- Does `GET /api/v1/members` return `committee_codes` (list of code strings for the selected year) per member? If not, add to `MemberSummary` schema to avoid per-row detail fetches for the COMMISSION column.
- Does `complete-registration` accept `committee_uuids` to atomically assign committees at activation, or should committee roster assignment be a separate sequential call?
- Does the Committee model expose `last_meeting_date` and `budget_status` fields for the Administrative Overview table?

### Excluded scope
- New backend aggregate endpoints for KPI (deferred).
- Photo upload / avatar images (initials only in V2).
- Backend business rule changes beyond existing `complete-registration` and committee roster contracts.
